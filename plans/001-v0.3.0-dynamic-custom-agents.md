## Why this matters

v0.3.0 unifies both hosts on `parent → coordinator background session → dynamic,
project-scoped custom expert subagents` (ADR 0001). On Claude this is **required**
by host behavior: custom agents load at *session start*, so a nested sub-agent in
the running parent cannot see `.claude/agents/*.md` the parent just wrote. Only a
**fresh background session** (`claude --bg`) re-scans `.claude/agents/` and can
spawn the projected experts. This replaces the v0.2.x nested-orchestrator shape.

## Topology (Claude instantiation)

```
parent skill (user's session, thin)
  collect brief
  derive per-topic experts → .claude/agents/swarm-<runId>-<role>.md  (from agents/swarm-expert.md template + runtime persona plan)
  record roster: {projectedName, projectedPath, projectedSha256, persona}
  claude --bg --agent swarm-discussion:swarm-coordinator --name swarm-<runId> "<packet>"   (capture short id)
  poll: claude agents --json / claude logs <id>  until terminal index
  relay synthesis + artifact paths only
  claude stop <id> ; claude rm <id> ; delete the roster files it created   (on every terminal/failure path)

coordinator background session (fresh → sees the projected agents)
  resolve wrapper/runtime → init → context-build → prompt-build (per expert)
  spawn each projected expert as a depth-1 subagent (Agent tool) with the runtime prompt.txt ONLY
  format each result into a wait-batch → transport-append-batch → transport-collect
  write spawn-order WITH agentDescriptor + host-step customAgentProjection
  append-message → checkpoint → finalize-round → validate-host-step/validate-loop/validate-discussion → trace → evidence
  return ONLY a compact terminal index

expert (.claude/agents/swarm-<runId>-<role>.md — no-tools leaf, Read only)
  receive one runtime prompt → return only the requested JSON → stop
```

`agentDescriptor` (runtime-owned; written by the coordinator via the runtime
transport CLI): `projectedName`, `projectedPath`, `projectedSha256`, `agentType`
(= projectedName), `invocationForm: "explicit_spawn"`, `promptRef`. Claude
`resultKey` is `name`; the host id recorded is the projected agent name/instance.

## Steps

1. **Settings: disable bg worktree isolation** (ADR Risk R2). Add
   `.claude/settings.json` with `{"worktree": {"bgIsolation": "none"}}` so the
   coordinator background session writes the discussion artifact tree to the real
   checkout, not `.claude/worktrees/`. (Alternative: pass an absolute
   `discussionDir` in the packet and have the coordinator assert it.)

2. **Expert template** (`agents/swarm-expert.md`). Keep it as the canonical
   no-tools leaf (Read only; returns only the requested JSON; never spawns).
   This file is the **template** the parent specializes per-topic. Document the
   projection rule: derived file `name: swarm-<runId>-<roleSlug>`, body = a thin
   persona shell that defers all phase content to the runtime-produced prompt.

3. **Coordinator agent**: rename `agents/swarm-orchestrator.md` →
   `agents/swarm-coordinator.md` (unified vocabulary with Codex). Tools:
   `Bash, Read, Agent`. Contract: runtime loop only; spawn the **projected**
   expert subagents by name with the runtime `prompt.txt`; write spawn-order with
   `agentDescriptor` and host-step `customAgentProjection` via the runtime CLI;
   persist each result through `transport-append-batch` before merge; never
   create/edit/delete `.claude/agents/` files; return only the compact index.
   It must NOT carry Write of discussion artifacts directly — the vendored runtime
   CLI (via Bash) owns all artifact writes.

4. **Parent skill** (`skills/swarm-discussion/SKILL.md`): rewrite for the new
   lifecycle — collect brief; derive + write the roster `.claude/agents/swarm-<runId>-*.md`
   (collision-safe names; record path + sha256 + persona); dispatch ONE
   coordinator background session (`claude --bg --agent swarm-discussion:swarm-coordinator
   --name swarm-<runId>`); poll `claude agents --json` / `claude logs <id>` to the
   terminal index; relay synthesis + paths; then `claude stop <id>`, `claude rm <id>`,
   and delete only the roster files it created. Cleanup must run on success,
   failure, and coordinator-creation failure; defer cleanup if the coordinator
   may still be running (timeout) until a terminal result or explicit abandonment.
   Boundaries: parent never spawns experts, never runs runtime mechanics, never
   writes discussion artifacts, never deletes user/plugin agents.

5. **Wrapper `doctor` capability probe** (`bin/swarm_runtime_wrapper.py`): add a
   background-session probe (`claude agents --json` / `claude --bg` reachable?)
   and record `hostBackgroundSessions.supported`; keep the nesting probe. `doctor`
   must FAIL FAST with a clear capability error when background sessions are
   unavailable (D2 is background-session-only — no silent fallback).

6. **Governance**: update `AGENTS.md` — principle #2 (orchestrator sub-agent →
   coordinator background session), rename orchestrator→coordinator everywhere,
   add the background-session dependency + minimum Claude Code version
   (nesting ≥ v2.1.172; agent view ≥ v2.1.139 — confirm the floor here) + the
   `worktree.bgIsolation` requirement. Bump `.claude-plugin/plugin.json` to `0.3.0`.

7. **Re-vendor** (only after runtime 007+008 land): re-vendor at the v0.3.0
   runtime commit with the runtime repo's `scripts/vendor.py`, then
   `scripts/vendor.py verify`. Run `doctor --smoke-fixture`.

8. **Retained projected smoke**: drive ONE real Claude discussion through the new
   topology (real projected `.claude/agents/`, real coordinator background
   session, real expert subagents). Retain the full tree under
   `smoke/discussions/<id>` with `agentDescriptor`s in spawn-order,
   `customAgentProjection` in host-step, descriptors in collect-result, and the
   prompt artifacts at the referenced `promptRef`s.

9. **Certify**: from the runtime repo,
   `conformance/certify_adapter.py --discussion <repo>/smoke/discussions/<id> --vendored <repo>/vendor/swarm-runtime --runtime <repo>/vendor/swarm-runtime/runtime/swarm_rt.py`
   → CERTIFIED, including the plan-008 projected-fan-out gate (proves real
   projected fan-out, not the generic path).

10. **Release**: `README.md` release notes state the pinned `runtimeSha`,
    compatibility string, minimum Claude Code version, and the background-session
    dependency. Tag `v0.3.0`. (Aggregator pin bump happens in `swarm-discussion`
    after both adapters certify.)

## STOP conditions

- Background sessions are unavailable/disabled in the target environment →
  STOP at step 5 and report; v0.3.0 has no nested fallback (D2). Do not ship.
- The coordinator background session writes artifacts into `.claude/worktrees/`
  (step 1 not effective) → STOP; fix isolation before the smoke.
- Certification can only pass against the vendored fixture, not the retained real
  projected discussion → STOP (repeat of the v0.2.1 review finding; the real tree
  must be retained and certified).
- Any projected `.claude/agents/` file survives a completed run → STOP; cleanup
  is a hard contract.

## Notes / risks (see ADR 0001)

- R1 background-session research-preview dependency is accepted (D2). Pin the min
  version and fail fast in `doctor`.
- R3 run-scoped names; never touch user/plugin agents; guaranteed cleanup.
- R5 the coordinator uses unrestricted `Agent` (dynamic expert names defeat a
  static `Agent(types)` allowlist); safety rests on the coordinator contract and
  the experts being no-tools leaves.
- R7 cost: N expert subagents + 1 coordinator background session per discussion.
